Corona in Bangladesh - Five Lessons From Our First Public Website

April 27, 2020 5-minute read
advice • backend • web • node • google cloud • heroku • express • coronainbangladesh

Last month we launched our website coronainbangladesh. The response was pretty great. In the span of a month, we’ve received over 50,000 visitors, news coverage and an acknowledgemet in the official government website. However, having such a high amount of traffic comes with it’s own set of challenges. I’ve had previous experience developing production systems, but those were mainly Bots, Business integrations and CMS sites which don’t receive a fraction of the traffic.

This article is a series of lessons I learnt throughout the development, updates and maintainance of coronainbangladesh. While reading, do keep in mind this guide is meant for novices and is written by one.
So let’s begin

Lesson 1 - Use a Platform as a Service

Image Source
If you’re a beginner and don’t know what’s the best way to get your code into a live production server, using a PaaS offering like Heroku or Google App Engine will make things much easier. These platforms are “fully managed” meaning they do much of the heavy lifting for you. They come built in with features like

  • Simplified deployments - Deploy with just one command and with zero downtime
  • Automatic scaling - Platform starts and closes new instances on the fly depending on traffic
  • Live staging server - Test your code in a live “production” environment before serving to the the pubic

Now PaaS aren’t a silver bullet, they do have their own set of problems. They usually cost more and aren’t as easily customizable as a conventionnal VM. You sometimes have to write code in your application specific to the PaaS you’re using. However, they can serve as a really useful set of training wheels when you’re still learning the ropes.

Lesson 2 - Code like you’re being attacked (because you probably are)

We had our first DDos attack launched against us on day 5. I have no idea why anyone would want to attack something as benign as a our website, but they did. The chances of your website being attacked or exploited increases with the amount of traffic it receives. So security should be a first priority from the very beginning of development, not as an afterthought.

Make sure you do the basics - Securing your API endpoints, using SSL, hashing all Passwords, turning on CORS etc. You can find great framework specific security best practises all over the internet. Go through them before going into production. We followed this checklist on the express official website.

Lesson 3 - Constantly monitor your server resources

Keep an eye out for how much Memory, CPU, Bandwidth your system is consuming. These metrics are an important indicator of the overall health of your system. Anomalies in any of these can be due to bugs, memory leaks or other performance issues. This can also give you an idea about resource utilization and help you decide to scale up or down a specific resouce. If you’re using some fancy cloud service (Remember Lesson 1?), then a lot of these monitoring tools should come built in.

Additionally, make sure to clean up resources you’re not using or you might find a nasty surprise waiting for you in your monthly bill. In our case, I made this totally stupid move where I forgot to disconnect from the database when my process instance was terminating. This resulted in about 300 concurrent connections to our database server just from localhost….yikes

Lesson 4 - Keep the scope of your project in mind

Decision you make should always consider the scope and scale of your project. Just because something make’s sense for Amazon does not mean it will also make sense for you. There is no one size fits all solution and over engineering is a genuine problem.

In our case, when we needed to add a cache to our server, we sort of unanimously decided we should use redis. This was simply because that’s what we all read we should use from the internet. However, the absolute minimum sized configuration of redis on our cloud provider was about 100 times larger than we needed and costed more than our server and database combined. It was ridiculous, we could fit our entire database on the redis server 100 times over and have no problem. Finally after rolling with redis for a few days, we decided to ditch it completely and just use a super simple in-memory caching library from npm. This worked perfectly, and it was free.

Lesson 5 - Run a lot of Tests

Confession: I don’t like writing tests. It’s painstaking, repetetive and utterly boring. So I didn’t write any tests when we launched our website. And…. I got screwed. Weird bugs were popping up, dependencies were breaking and all of these problems expressed themselves in the most unsuspecting of ways. The source of the bug is never in the first place you look. I had to spend hours chasing down trivial issues that would have been immidiately apparent if I had a comprehensive test suite.

I understand you’re working with limited time and resources. But what you must realized is, writing tests actually saves you time in the long run. It’s an investment that ends up paying massive dividends. But if you really don’t have time to write a comprehensive test suite, prioritize and start with API (Component) testing. This is the easiest way to write and provides more coverage than unit testing (you may even craft API tests without code using tools like Postman).

Conclusion

People learn best from experience. In that sense, this whole project has been a wonderful learning experience. Want to become a better developer? Go out there, make beautiful random things, see them break and then fix them. Trust me, that’s the way your learning will have the most impact.
